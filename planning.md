# 블로그 HTML 템플릿 생성 프로그램 기획서

## 1. 프로젝트 개요

### 1.1 프로젝트 목적

사용자가 입력한 일반 텍스트를 HTML 형식으로 자동 변환하고, 선택한 템플릿 종류와 블로그 플랫폼에 맞는 HTML 템플릿 파일을 결합하여 최종 HTML 결과를 생성하는 로컬 프로그램을 개발한다.

본 프로그램의 주요 사용자는 HTML에 대한 전문 지식이 없는 비개발자이다.

따라서 일반적인 콘텐츠 작성 과정에서는 사용자가 HTML을 직접 작성하거나 수정할 필요가 없어야 한다.

단, 템플릿 설정 과정에서는 시각적 에디터와 HTML 편집 기능을 모두 제공하여 필요한 경우 사용자가 편집 가능한 영역을 직접 수정할 수 있도록 한다.

---

# 2. 프로젝트 핵심 원칙

* 최대한 단순한 구조로 구현한다.
* 불필요한 기능을 추가하지 않는다.
* 한 페이지 UI를 사용한다.
* 별도의 변환 버튼을 사용하지 않는다.
* 입력 및 선택 변경 사항은 실시간으로 결과에 반영한다.
* 프로그램은 로컬 환경에서 실행한다.
* 서버를 사용하지 않는다.
* 데이터베이스를 사용하지 않는다.
* AI 기능을 사용하지 않는다.
* 템플릿 HTML 파일은 별도의 폴더에서 관리한다.
* 템플릿 종류와 플랫폼을 기준으로 필요한 HTML 파일을 직접 불러온다.
* 템플릿 파일 경로 관리를 위해 JSON 설정 파일을 사용하지 않는다.
* 기존 `config/settings.json` 파일은 절대 수정하지 않는다.
* 사용자가 수정할 수 있는 템플릿 영역은 각 HTML 파일 내부에 정확히 한 곳만 존재한다.
* 템플릿 저장 시 선택된 HTML 파일의 편집 가능 영역만 수정한다.

---

# 3. 주요 기능

프로그램의 핵심 기능은 다음과 같다.

1. 템플릿 종류 선택
2. 일반 텍스트 입력
3. 입력 텍스트의 실시간 HTML 변환
4. 블로그 플랫폼 선택
5. 선택된 HTML 템플릿 파일 로드
6. 입력 HTML과 템플릿 HTML 결합
7. 최종 결과 실시간 생성
8. 미리보기
9. 에디터 보기 및 수정
10. HTML 보기 및 수정
11. 템플릿 수정 내용 복구
12. 선택된 템플릿 저장

# 5. 지원 플랫폼

| 표시 이름 | 내부 값        |
| ----- | ----------- |
| 네이버   | `naver`     |
| 워드프레스 | `wordpress` |
| 자사몰   | `company`   |

플랫폼 선택 영역은 반드시 내용 입력 영역 아래에 배치한다.

사용자는 하나의 제품 내용을 작성한 후 플랫폼을 변경하면서 각 플랫폼에 맞는 결과를 확인한다.

플랫폼을 변경해도 입력한 내용은 유지한다.

# 10. 텍스트 입력

사용자는 일반 텍스트 형식으로 제품 정보를 입력한다.

예:

```text
기업 : 다나와
품명 : bio 비닐
규격 : 320 × 430
소재 : 생분해 비닐
인쇄 : 단면 1도
```

사용자는 일반 콘텐츠 작성 과정에서 HTML을 입력하지 않는다.

---

# 11. 텍스트 HTML 변환

사용자가 입력한 일반 텍스트는 실시간으로 HTML로 변환한다.

입력:

```text
기업 : 다나와
품명 : bio 비닐
규격 : 320 × 430
소재 : 생분해 비닐
```

변환:

```html
<p><strong>기업 :</strong> 다나와</p>
<p><strong>품명 :</strong> bio 비닐</p>
<p><strong>규격 :</strong> 320 × 430</p>
<p><strong>소재 :</strong> 생분해 비닐</p>
```

---

# 12. 텍스트 변환 규칙

## 12.1 `:` 문자가 존재하는 경우

각 줄에서 처음 등장하는 `:` 문자를 기준으로 라벨과 값을 분리한다.

입력:

```text
기업 : 다나와
```

변환:

```html
<p><strong>기업 :</strong> 다나와</p>
```

`:` 앞부분은 `<strong>` 태그를 적용한다.

---

## 12.2 `:` 문자가 존재하지 않는 경우

일반 문단으로 처리한다.

입력:

```text
친환경 소재를 사용한 제품입니다.
```

변환:

```html
<p>친환경 소재를 사용한 제품입니다.</p>
```

---

## 12.3 빈 줄

빈 줄은 HTML 문단 구분으로 처리한다.

예:

```html
<p><br></p>
```

---

## 12.4 HTML Escape

사용자 입력 문자열은 HTML Escape 처리를 수행한다.

주요 처리 대상:

```text
&
<
>
"
'
```

사용자 입력으로 인해 HTML 구조가 손상되지 않도록 한다.

# 13. 템플릿 파일 관리

템플릿 HTML 파일은 별도의 폴더에서 관리한다.

```text
templates/
├── smallQuantity-naver.html
├── smallQuantity-wordpress.html
├── smallQuantity-company.html
├── custom-naver.html
├── custom-wordpress.html
└── custom-company.html
```

총 6개의 HTML 파일을 사용한다.

---

# 14. 템플릿 파일 선택 규칙

선택된 템플릿 종류와 플랫폼 값을 조합하여 파일명을 생성한다.

규칙:

```text
{templateType}-{platform}.html
```

예:

```text
smallQuantity
+
naver

↓

smallQuantity-naver.html
```

다른 예:

```text
custom
+
wordpress

↓

custom-wordpress.html
```

템플릿 경로를 확인하기 위해 JSON 설정 파일을 사용하지 않는다.

---

# 15. settings.json 정책

기존 프로젝트에 다음 파일이 존재할 수 있다.

```text
config/settings.json
```

해당 파일은 절대 수정하지 않는다.

금지 사항:

* 파일 수정
* 파일 저장
* 파일 구조 변경
* 파일 삭제

현재 프로그램의 템플릿 선택 및 저장 기능에서 `settings.json`을 사용하지 않는다.

---

# 16. 템플릿 HTML 구조

각 템플릿에는 편집 가능한 영역이 정확히 한 곳 존재한다.

기본 구조:

```html
<header>
  고정 영역
</header>

<!-- EDITABLE_START -->

<section>
  사용자가 수정할 수 있는 영역
</section>

<!-- EDITABLE_END -->

<footer>
  고정 영역
</footer>
```

---

# 17. 편집 가능 영역 정책

사용자는 다음 영역만 수정할 수 있다.

```text
EDITABLE_START

~

EDITABLE_END
```

다음 영역은 수정할 수 없다.

* `EDITABLE_START` 이전 영역
* `EDITABLE_END` 이후 영역
* `EDITABLE_START` 마커
* `EDITABLE_END` 마커
* 템플릿 외부 구조
* 플랫폼별 고정 HTML 구조

편집 가능 영역의 길이에는 제한을 두지 않는다.

---

# 18. 최종 HTML 생성

일반 콘텐츠 작성 시 최종 HTML은 다음 두 요소를 결합한다.

```text
HTML로 변환된 사용자 입력

+

선택된 템플릿 HTML 전체
```

예:

```html
<section class="product-information">
  <p><strong>기업 :</strong> 다나와</p>
  <p><strong>품명 :</strong> bio 비닐</p>
</section>

<header>
  고정 영역
</header>

<!-- EDITABLE_START -->

<section>
  템플릿 내용
</section>

<!-- EDITABLE_END -->

<footer>
  고정 영역
</footer>
```

현재 기획에서는 사용자 입력 HTML 아래에 선택된 템플릿 HTML 전체를 연결한다.

# 19. 결과 표시 모드

결과 영역은 다음 세 가지 모드를 지원한다.

| 모드      | 역할              | 수정 가능 여부 |
| ------- | --------------- | -------- |
| 미리보기    | 실제 렌더링 결과 확인    | 불가       |
| 에디터 보기  | 시각적 콘텐츠 및 서식 수정 | 조건부 가능   |
| HTML 보기 | HTML 코드 확인 및 수정 | 조건부 가능   |

---

# 20. 미리보기

미리보기에서는 현재 생성된 최종 HTML 결과를 실제 렌더링 형태로 표시한다.

사용자는 미리보기 내용을 직접 수정할 수 없다.

예상 구현 방식:

```text
iframe
```

최종 HTML을 `iframe.srcdoc`을 통해 표시할 수 있다.

---

# 21. 에디터 보기

에디터 보기에서는 시각적 에디터를 사용한다.

예상 에디터:

```text
CKEditor
```

사용자는 HTML을 직접 다루지 않고 편집 가능 영역의 다음 요소를 수정할 수 있다.

* 텍스트
* 문단
* 굵기
* 기타 지원되는 서식

에디터 보기에서는 전체 HTML 템플릿을 수정하지 않는다.

`EDITABLE_START`와 `EDITABLE_END` 사이의 내용만 수정한다.

---

# 22. HTML 보기

HTML 보기에서는 편집 가능 영역의 HTML 코드를 표시한다.

전체 템플릿 HTML은 수정 대상으로 제공하지 않는다.

HTML 보기에서 사용자는 편집 가능 영역의 코드를 직접 수정할 수 있다.

표시 대상:

```text
EDITABLE_START와 EDITABLE_END 사이의 HTML
```

HTML 보기에서 마커 자체는 표시하거나 수정하지 않는다.

---

# 23. 에디터 보기와 HTML 보기 동기화

에디터 보기와 HTML 보기는 동일한 편집 데이터를 공유한다.

## 에디터 → HTML

```text
에디터 수정

↓

현재 HTML 데이터 추출

↓

HTML 보기 갱신

↓

미리보기 갱신
```

## HTML → 에디터

```text
HTML 수정

↓

HTML 유효성 검사

↓

에디터 갱신

↓

미리보기 갱신
```

---

# 24. 템플릿 수정 가능 조건

사용자 내용 입력 영역이 비어 있을 때만 템플릿을 수정할 수 있다.

## 내용이 존재하는 경우

* 미리보기 활성화
* HTML 보기 활성화
* HTML 보기는 읽기 전용
* 에디터 수정 비활성화
* 복구 비활성화
* 저장 비활성화

## 내용이 비어 있는 경우

* 미리보기 활성화
* 에디터 수정 활성화
* HTML 수정 활성화
* 복구 조건부 활성화
* 저장 조건부 활성화

---

# 25. 복구 기능

결과 영역 하단에 다음 버튼을 제공한다.

```text
[복구] [저장]
```

복구 버튼은 저장 버튼 왼쪽에 배치한다.

---

# 26. 복구 기준

템플릿을 처음 불러오면 현재 편집 가능 영역의 HTML을 복구 기준으로 저장한다.

개념적 상태:

```ts
originalEditableHtml
currentEditableHtml
```

템플릿 로드:

```text
originalEditableHtml = 불러온 편집 영역
currentEditableHtml = 불러온 편집 영역
```

---

# 27. 복구 동작

사용자가 복구 버튼을 누르면 마지막으로 불러왔거나 저장한 상태로 돌아간다.

처리:

```text
복구 클릭

↓

currentEditableHtml을 originalEditableHtml로 변경

↓

에디터 보기 갱신

↓

HTML 보기 갱신

↓

미리보기 갱신
```

실제 HTML 파일은 변경하지 않는다.

---

# 28. 저장 후 복구 기준

저장이 성공하면 저장된 내용을 새로운 복구 기준으로 설정한다.

```text
템플릿 저장 성공

↓

originalEditableHtml = currentEditableHtml
```

이후 다시 수정하고 복구 버튼을 누르면 가장 최근에 저장된 상태로 돌아간다.

복구 기능은 프로그램 최초 설치 상태로 돌아가는 기능이 아니다.

---

# 29. 복구 버튼 활성화 조건

다음 조건을 만족할 때 복구 버튼을 활성화한다.

```text
내용 입력 영역이 비어 있음

AND

템플릿 선택 완료

AND

플랫폼 선택 완료

AND

현재 편집 내용이 복구 기준과 다름
```

---

# 30. 저장 기능

저장 버튼을 누르면 현재 선택된 HTML 파일 하나만 수정한다.

예:

```text
템플릿:

smallQuantity

플랫폼:

naver
```

저장 대상:

```text
smallQuantity-naver.html
```

다른 5개의 HTML 파일은 수정하지 않는다.

---

# 31. 저장 방식

HTML 파일 전체를 에디터 결과로 덮어쓰지 않는다.

원본 파일에서 다음 영역만 교체한다.

```text
EDITABLE_START

~

EDITABLE_END
```

처리:

```text
원본 HTML 파일 로드

↓

EDITABLE_START 위치 확인

↓

EDITABLE_END 위치 확인

↓

두 마커 사이 내용 교체

↓

선택된 HTML 파일 저장
```

---

# 32. 저장 버튼 활성화 조건

다음 조건을 모두 만족해야 저장 버튼을 활성화한다.

```text
내용 입력 영역이 비어 있음

AND

템플릿 선택 완료

AND

플랫폼 선택 완료

AND

편집 내용이 비어 있지 않음

AND

현재 편집 내용이 복구 기준과 다름

AND

HTML 검증 통과
```

---

# 33. HTML 검증

HTML 보기에서 수정된 내용은 저장 전 검증한다.

최소 검증 대상:

* 편집 내용이 비어 있는지 확인
* `EDITABLE_START` 문자열 삽입 여부
* `EDITABLE_END` 문자열 삽입 여부
* `<script>` 태그 포함 여부
* 위험한 이벤트 속성 포함 여부
* 기본적인 HTML 구조 오류 여부

검증 실패 시 저장하지 않는다.

---

# 34. 템플릿 마커 검증

각 HTML 파일에는 다음 마커가 정확히 하나씩 존재해야 한다.

```text
EDITABLE_START
EDITABLE_END
```

검증 조건:

1. `EDITABLE_START` 정확히 한 개
2. `EDITABLE_END` 정확히 한 개
3. `EDITABLE_START`가 `EDITABLE_END`보다 앞에 존재

조건을 만족하지 않으면 템플릿 수정 및 저장을 진행하지 않는다.

---

# 35. 템플릿 또는 플랫폼 변경

사용자가 템플릿 종류 또는 플랫폼을 변경해도 입력 내용은 유지한다.

예:

```text

↓

내용 입력

↓

네이버 선택

↓

워드프레스 선택

↓

입력 내용 유지
```

---

# 36. 저장되지 않은 템플릿 수정 사항 처리

내용 입력이 비어 있는 템플릿 수정 상태에서 저장하지 않은 변경 사항이 존재할 수 있다.

이 상태에서 다른 템플릿 또는 플랫폼을 선택하면 확인창을 표시한다.

예:

```text
저장하지 않은 템플릿 변경 사항이 있습니다.

변경 내용을 취소하고 이동하시겠습니까?
```

선택:

```text
[계속 수정] [변경 취소 후 이동]
```

# 37. 실시간 처리 정책

본 프로그램의 작업은 비교적 연산량이 작으므로 실시간 처리를 기본으로 한다.

별도의 변환 버튼을 사용하지 않는다.

텍스트 입력의 경우 짧은 Debounce를 적용할 수 있다.

권장 예:

```text
100ms ~ 200ms
```

기본 권장값:

```text
150ms
```

사용자 입장에서는 실시간으로 동작하지만 불필요한 반복 렌더링을 줄일 수 있다.

템플릿 종류 및 플랫폼 선택은 즉시 반영한다.

# 47. 최종 기능 정의

1. 사용자는 소량제작 또는 맞춤제작 템플릿을 선택한다.

2. 사용자는 일반 텍스트로 제품 정보를 입력한다.

3. 프로그램은 입력 내용을 실시간으로 HTML로 변환한다.

4. 사용자는 네이버, 워드프레스, 자사몰 중 하나를 선택한다.

5. 프로그램은 선택값에 맞는 HTML 템플릿 파일을 직접 불러온다.

6. 프로그램은 변환된 입력 HTML과 템플릿 HTML 전체를 결합한다.

7. 결과는 별도의 변환 버튼 없이 실시간으로 갱신된다.

8. 사용자는 미리보기, 에디터 보기, HTML 보기 사이를 전환할 수 있다.

9. 내용 입력이 존재하면 에디터 및 HTML 수정 기능을 사용할 수 없다.

10. 내용 입력이 비어 있으면 선택된 템플릿의 편집 가능 영역을 수정할 수 있다.

11. 편집 가능 영역은 각 HTML 템플릿에 정확히 한 곳만 존재한다.

12. 에디터 보기와 HTML 보기 모두 편집 가능 영역만 수정한다.

13. 에디터 보기와 HTML 보기는 동일한 편집 데이터를 공유한다.

14. 수정 내용은 실시간으로 미리보기에 반영된다.

15. 복구 버튼을 누르면 마지막으로 불러왔거나 저장한 상태로 돌아간다.

16. 저장 버튼을 누르면 현재 선택된 HTML 파일의 편집 가능 영역만 변경한다.

17. 다른 HTML 템플릿 파일은 수정하지 않는다.

18. 저장 성공 후 저장된 내용을 새로운 복구 기준으로 설정한다.

19. `config/settings.json`은 절대 수정하지 않는다.

20. 프로그램은 한 페이지 구조와 최소 기능 원칙을 유지한다.
